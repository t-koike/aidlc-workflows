# Changelog

All notable changes to this project will be documented in this file.

## [2.5.3] - 2026-07-17

AI-DLC now has a checksum-verified, self-contained macOS/Linux install channel with project initialization, retained versions, project pins, upgrade, rollback, and offline packages. Existing copy installs remain supported and unchanged in how they are invoked. **Upgrade:** binary users run `install.sh --harness <name>` and then `aidlc init`; copy-install users may continue copying `dist/<harness>/` and running through bun.

* `aidlc init` initializes or refreshes a complete harness projection locally, supports `--dry-run`, exact apply approval with `--plan-token`, `--force`, `--from`, `--harness`, and `--mcp defaults|none`, preserves project-owned method and plugin-composed data, and merges declared root integrations without replacing unrelated content.
* `aidlc upgrade`, `aidlc rollback`, `aidlc use`, `aidlc versions list|install`, and `aidlc package create|verify` provide transactional machine lifecycle, exact side-by-side versions, commit-ready project pins, and air-gapped release sets.
* Release assets now include one binary per target, one binary-invocation data archive per harness, `install.sh`, `version.json`, and `checksums.txt`; the checksum set authenticates the manifest before its asset metadata is trusted, and the tag workflow publishes and attests the resulting assets.
* Release CI executes both musl binaries inside matching Alpine containers before publication instead of treating cross-target format inspection as a smoke run.
* Install and lifecycle mutations use one lean transaction core with same-filesystem staging, atomic rename-over pointer changes, in-process snapshot rollback, and lazy abandoned-staging cleanup; no startup recovery journal is required on macOS/Linux.
* Release projections carry native host trust entries for Claude, Kiro CLI, Kiro IDE, and Codex. `install.sh` supports channel-aware download progress, human, quiet, and schema-versioned JSON output plus opt-in transactional PATH profile blocks through `--profile <startup-file>`; it refuses package-manager-owned and mixed-ownership command destinations.
* Every distribution carries schema-versioned harness identity, projection policy, and version stamps. `dist-release/<harness>/` is committed and drift-guarded beside the existing copy-install `dist/<harness>/`.
* Public lifecycle exits are disjoint: 0 success, 1 operational failure, 2 usage, 3 unavailable network result, 4 integrity/safety refusal, and 5 action needed.
* `aidlc doctor --json` reports the same check set as human output, treats bun as optional under the compiled binary, and diagnoses installed-runtime, transaction staging, pin registry, project-stamp, and native host-trust state.
* Breaking for CI/scripts: release tooling must package `dist-release/` with `bun scripts/package-release.ts`; do not publish `build/binaries/*/runtime/` as standalone release data.

## [2.5.2] - 2026-07-21

`/aidlc --doctor` gains an `--export` flag that writes a small, redacted diagnostic report so a misbehaving workflow can be debugged without sharing the whole project directory. One fresh doctor analysis backs both the live `--doctor` render and the export; the workflow diagnosis is advisory (it never changes the exit code — only the existing environment/config checks do, preserving the behaviour CI scripts gate on), and the export merges the legacy environment findings so `report.md`/`report.json` carry the same findings the live report shows. It reconstructs the workflow timeline from the audit trail (stage durations, gates, revisions — scoped to the latest run, events timestamp-sorted, repeated attempts paired chronologically), runs deterministic condition→remedy rules (unresolved gates, state/audit drift, stale/missing runtime graph gated on a workflow existing, cold hooks), and exports only allowlisted, normalized fields — never artifact, contribution, question, or memory bodies. Every project/home path and every custom intent/stage/agent (including `lead_agent`) identifier is hashed or redacted before serialization; secrets are scrubbed in plain, JSON-quoted, JSON-escaped, and punctuation-bearing forms; inputs whose real path escapes the project root are refused (leaf or parent symlink), platform-separator aware for Windows; a malformed runtime graph degrades to a finding instead of crashing doctor; truncation keeps the JSON artifacts valid; per-file and total size are capped. **Upgrade:** re-copy your `dist/<harness>/` shell into the project.

* `/aidlc --doctor --export` runs a fresh doctor pass and writes a report directory (`report.md`, `report.json`, `manifest.json`, and normalized `evidence/`) under `aidlc/diagnostics/` (gitignored on all five harnesses), packaged as a timestamped `aidlc-diagnostic-report-<ts>.tar.gz` when a system `tar` is available and left as a directory (with manual-share instructions) otherwise. Override the output location with `--output <dir>`; a bare `--output` with no path errors instead of creating a directory named `true`. The export surface routes through both terminal-command deciders (`parseNextFlags` and `classifyTerminalCommand`), so it reaches the tool on Claude Code, Kiro CLI/IDE, and Codex, not only a direct invocation.
* Plain `/aidlc --doctor` also surfaces the new structured findings (`gate-unresolved`, `runtime-graph-stale`, and the rest) as an **advisory** section that never changes the exit code.
* Diagnosis is deterministic (no LLM) and recovery bypasses (any `AIDLC_DISABLE_*` env var or a workspace-reset instruction) are always reported as unsafe-to-automate with a prominent warning.
* No new package dependency is introduced; the export never includes workspace source, raw state/audit/runtime-graph files, artifact/contribution/question/memory bodies, environment variables, or command output.

## [2.5.1] - 2026-07-20

The adaptive composer now estimates implementation entropy before composing. It scores five components - intent ambiguity, codebase structural uncertainty, verification entropy, risk, and unresolved assumptions - from the task and the workspace, then composes the minimum viable workflow: the least sufficient EXECUTE/SKIP grid that still produces every artifact the outcome depends on. When CodeKB MCP tools are configured and the relevant spaces are indexed, the composer grounds its structural estimates in CodeKB call-graph and component analysis instead of scanning the workspace; without CodeKB it falls back to the bounded workspace scan. This is a prompt-only release - no new tools, schema fields, or engine changes. **Upgrade:** re-copy your `dist/<harness>/` shell into the project.

* `/aidlc compose` proposals now carry an entropy-score breakdown alongside the EXECUTE/SKIP grid: the gate renders three blocks - the validator's summary line, a five-component score table with bands and an advisory composite, and a per-stage decision table with reasoning for every EXECUTE and SKIP.
* Composed grids fold overlapping ideation/inception stages by default (each fold names its un-SKIP trigger); the human still approves, edits, or rejects every proposal at the gate.
* In-flight recompose proposals re-estimate the entropy components from what completed stages actually resolved, name that evidence per proposed flip, and strict-validate before the gate so a starved flip is caught before approval, not after.
* With CodeKB evidence covering the affected codebase, the composer may propose skipping Reverse Engineering; the proposal must disclose that downstream stages then run without the local RE artifact store, and the human decides at the gate. CodeKB is an optional external MCP server (see the Scopes and Depth guide for per-harness setup); without it the composer scores from the workspace scan, and on Kiro CLI / Kiro IDE the fallback is always used.

## [2.5.0] - 2026-07-17

The three-role ensemble (North Star Goal 1): a stage's `mode` is now its communication topology - who talks to whom while the body runs - and support agents become real, independently dispatched collaborators instead of voices the conductor adopts. Two new modes join `inline` and `subagent`: `pipeline` (chain: the links collectively author the artifacts in declared order, each seeing all upstream work; the final link leaves them complete) and `mob` (mesh as bounded rounds: all support agents contribute in parallel against the lead's draft, mutually blind; unresolved objections are triaged - judgment calls surface to the human mid-stage, knowledge disputes get one confirm-or-maintain round; maintained dissent is quoted verbatim at the approval gate). `mode: subagent` is strengthened to hub-and-spoke: on a stage with `support_agents`, each support agent is dispatched as a mutually-blind spoke against the lead's draft and the lead integrates. The writing model mirrors a real working session - everyone writes their own work: each dispatched collaborator writes a contribution file (`<stage>/contributions/<agent-slug>.md`, identity-marker first line, Contribution + Positions) and the lead alone edits the stage's `produces[]` artifacts. Those files double as deterministic completion evidence: the engine refuses approving a mob (or subagent-with-supports) stage while a declared collaborator's contribution file is missing. The conductor is the bus on every topology (only it delegates), and the reviewer loop is unchanged - a NOT-READY re-invokes the lead alone. `practices-discovery` ships as the hub-and-spoke showcase on greenfield and brownfield work (Pipeline & Deploy lead draft, mutually blind Quality + Developer + DevSecOps spokes, human interview, lead integration); `user-stories` ships as the mob showcase (Product Manager lead, Design + Developer + Quality collaborators, Product Lead reviewer). The shipped topology is 28 inline / 2 subagent / 1 pipeline / 1 mob. **Upgrade:** re-copy your `dist/<harness>/` shell into the project; on Kiro CLI/IDE also re-copy `.kiro/agents/` (nine new delegation-target agent configs plus an updated conductor `trustedAgents` list).

* New stage modes `pipeline` and `mob` (stage frontmatter + run-stage directive). Both require a non-empty `support_agents` - the compile rejects a chain with no links or a one-occupant room. `agent-team` stays reserved.
* Collaborator contribution files: dispatched support agents write `contributions/<agent-slug>.md` under the stage's record dir (first line `**Collaborator:** <agent-slug>` verbatim). `report --result approved` on an ensemble stage errors while one is missing or malformed, naming the gap and the fix; per-unit evidence follows only units the stage actually executes, and a no-DAG single iteration checks the ordinary stage directory. `AIDLC_DISABLE_ENSEMBLE_EVIDENCE=1` is the recovery escape hatch.
* Mob objection triage puts the human in the room: a judgment-call objection (both positions legitimate - scope, risk appetite, priority tradeoffs) surfaces mid-stage as a structured question; only knowledge disputes go to round 2. Under autonomous Construction the mid-stage turn is skipped - dissent is recorded and surfaces at the final-batch gate; it never halts the run.
* `reverse-engineering` re-modes from `subagent` to `pipeline`, recognizing the shape its body always had (developer scans, architect synthesizes and writes - a 2-link chain). No behavior change; the mode now tells the truth.
* `user-stories` flips to `mode: mob` and gains `aidlc-developer-agent` + `aidlc-quality-agent` as collaborators alongside `aidlc-design-agent`. Expect 3-6 additional bounded sub-agent dispatches on scopes that run it; each rides the collaborator's projected tier and the `tier_cap` clamp.
* `practices-discovery` flips to `mode: subagent` for both greenfield and brownfield runs: the Pipeline & Deploy lead drafts first, Quality + Developer + DevSecOps inspect it as mutually blind spokes, the human interview resolves gaps and judgment, and the lead integrates. Resume dispatches only missing spokes, and approval remains open until promotion into `aidlc/spaces/<active-space>/memory/{team,project}.md` succeeds.
* Stage lifecycle outcomes are report-owned: conductors report approval, rejection, revision, completion, or an explicit `skipped` result through `aidlc-orchestrate.ts`; the engine performs the atomic state/audit transition and routes onward. Direct `aidlc-state.ts` lifecycle mutations now fail with an engine-ownership error, so scripts must migrate those calls to `aidlc-orchestrate.ts report` (or the corresponding engine routing command). The PreToolUse redirect fires on Claude Code, Kiro CLI, Codex, and opencode (Kiro IDE relies on the CLI-level error - its hook payloads omit tool arguments). Conditional skips require the current stage slug and a nonblank reason.
* Kiro CLI/IDE delegation roster closed: all 14 personas are now dispatchable (`.kiro/agents/*.json` + conductor `trustedAgents`); the nine new collaborator configs carry read+shell plus an `aidlc/spaces/**`-scoped write grant (for their contribution files). Plugin composition requires the complete dispatch surface for every dispatched agent - agent-v1 JSON plus conductor trust registration on Kiro, an agent config TOML on Codex, a native `.opencode/agents/` subagent on opencode (a plugin's own shipped persona counts there when it projects cleanly); every harness rejects the reserved, runtime-less `agent-team` mode. If the installed parser is unavailable, Kiro/Codex/opencode accept only an explicitly inline, reviewer-free stage. Unsafe stages left by an older no-clobber compose are retained but reported as degraded instead of being silently grandfathered.
* Kiro CLI pre-dispatches unambiguous first-turn `compose`, routing/config flags, and fresh-workspace `--scope` calls with the exact recovered arguments; ambiguous active-workflow freeform still uses the turn-bound pre-tool guard. Kiro CLI/IDE pin the native `todo_list` schema (including `task_list_description`) and `subagent` crew schema, remap file-backed A-E/X choices to numbered chat options, and forward the exact human approval choice to `report`. Probing `aidlc-utility.ts intent-birth --help` is now read-only instead of accidentally birthing the fallback `poc` scope.
* Human checkpoints remain real turn stops: consolidated answer confirmation and per-unit Code Generation plan approval now persist a blank question-file answer before prompting, and the Stop hook resolves the exact active-unit question directory instead of forcing continuation or accepting another unit's stale question.
* `aidlc-graph compile` emits an `[advisory]` on stderr when the per-unit Construction build stage carries a mode other than `subagent` - that shape silently takes the stage off the autonomous swarm path.
* Scope decisions recorded: transports are conductor-simulated on all five harnesses in v1 (native nested dispatch and live-team rooms are documented follow-ups; Kiro CLI refuses nested delegation at runtime, so the hub is its permanent floor); the review loop stays the `reviewer` field, not a mode.
* Isolated runs now carry `single: true`, bypass the approval gate (`gate: false`), and expose no next stage. The conductor reports exactly once with `report --single --result completed`; the engine returns terminal `done` without routing the main workflow, surfacing learnings, or parking. `report --single` enforces the same ensemble evidence shape as the main workflow path, but an isolated per-unit stage has no main-workflow Unit context, so its collaborator files are checked at the ordinary stage-level contribution directory.
* Kiro CLI/IDE grants the Developer collaborator scoped write access to `aidlc/spaces/**`. Kiro composition now rejects plugin stages that would dispatch plugin-owned agents without a native agent-v1 configuration and `trustedAgents` registration, naming the stage, agent, and remediation in the compose drops log.
* The resume menu routes every choice: `report --result resumed` now returns a per-choice directive - resume continues from the checkpoint, redo names the `aidlc-jump.ts execute --direction redo` command, jump asks for the target then routes through `next --stage`, and start fresh routes through `next --new-intent`; an unrecognized answer errors with the accepted choices instead of silently resuming.
* Swarm convergence evidence is scoped to the stage attempt that produced it: each `SWARM_UNIT_CONVERGED` row now carries `Stage` + `Run floor` fields, and batch advance, the ensemble evidence gate, and the artifact guard count a row only when both match the current attempt - a late `finalize` retry against a prior attempt's worktree, or another swarm stage reusing the same unit names, no longer satisfies the current run. Rows written by earlier versions fail closed: an in-flight swarm crossing the upgrade re-fans its already-converged units (finalize's re-verify makes the re-run safe).
* A gate rejection on Practices Discovery invalidates the promotion receipt: `report --result approved` refuses until `practices-promote` re-runs after the rejection, so revised drafts can never ride an earlier affirmation.
* A rejection-driven revision that changes a `produces[]` artifact re-runs the stage's reviewer before the gate reopens (fresh `## Review` verdict; the learnings ritual still runs once per stage), and the learnings question is its own logged turn before the engine opens the gate - `QUESTION_ANSWERED` precedes `STAGE_AWAITING_APPROVAL` in the audit ledger on every harness.
* On opencode, a plugin agent file the native-roster projection cannot emit now drops alone with a per-file log line; previously one malformed persona aborted the plugin's entire compose (knowledge, sensors, tools, and the graph recompile were all skipped). A plugin persona whose name collides with an installed native agent no longer lets its stage compose with a dangling dispatch target.

## [2.4.0] - 2026-07-14

Reviewers are now verifiers: every declared review runs under an explicit **adversarial review contract**. The reviewer's job is to refute the artifact, not confirm it - it assumes defects exist and hunts for them, and READY is the verdict it fails to reach after trying to break the artifact, not the default it starts from. Findings must ground in machine-checkable evidence where it exists: the reviewer runs the validation tools its invocation lists (via shell) and checks the artifact against its acceptance criteria, its stage definition, and the consumed upstream contracts; a finding backed only by opinion is a suggestion, not grounds for NOT-READY. This is a prompt-only release - the contract is written once into stage-protocol §12a (so any future reviewer inherits it) with short domain-voiced restatements in the two reviewer personas; no new tools, schema fields, or agents. Two scope decisions are on record: "validate deterministically" is read as grounding in machine-checkable evidence that already exists (validation tools, tests, acceptance criteria, contracts), not as building new domain validators; and the self-healing budget requirement is read as iterations OR tokens, satisfied by the existing `reviewer_max_iterations` (default 2) with human escalation at exhaustion - no token budget is added. The other two verifier properties were already in place: reviewers project a different model class than producers by default (2.3.1 tiers), and budget exhaustion already escalates to the human with full context. **Upgrade:** re-copy your `dist/<harness>/` shell into the project.

## [0.5.20] - 2026-06-06

Ships the swarm's deterministic convergence **referee**, `aidlc-swarm.ts`. The swarm fires only under human-granted Construction autonomy inside a live `/aidlc` session, so the conductor (that session) owns the fan-out and the retry loop; the tool owns only what must be deterministic — the convergence verdict, the anti-tamper guard, the serialised merge-back, and the typed failure envelope. ## [0.2.0] - 2026-05-03

Breaking release. State file format is now v5 and cannot be migrated from v4 — in-flight workflows must be restarted:

    mv aidlc-docs aidlc-docs-v4-archive && /aidlc --init --force --scope <scope>

* Workflow `Status` values simplified to `Running` and `Completed`. Replaces `In Progress`, `Paused`, and `Waiting for Approval` — pause has no UX and approval waits moved to the stage level (see next bullet)
* Two new stage states: `[?]` means the stage is waiting for your approval, `[R]` means you rejected the gate and it's being revised. Previously both situations looked like `[-]` and you couldn't tell them apart from `--status`
* `/aidlc --status` now tells you who the blocker is. When a stage is `[?]`: "Awaiting your approval on <stage>". When `[R]`: "Revising <stage> (revision N of 3)". Previously always read "Running" even when the workflow was waiting on you
* `/aidlc --doctor` gains a drift check — catches the rare case where the audit log records a workflow completion but the state file disagrees
* `/aidlc --init` is now one fast deterministic call (under a second). Replaces the previous multi-tool-call flow that sometimes hit timeouts
* `--force` flag on `/aidlc --init` — rewrites `aidlc-state.md` and warns about any unrelated artifacts found in `aidlc-docs/`. Without `--force`, re-running init on an existing workflow exits with a helpful message instead of hanging
* Approving a stage now advances the workflow atomically. The approval flips the gate, marks the stage complete, and moves you to the next stage in one step. Fixes the previous failure mode where `/aidlc --status` could show "Awaiting your approval" after you'd already approved
* Compaction-aware resume. If your Claude Code context is compacted during a stage, resuming surfaces a three-option prompt (continue / review prior artifact / restart stage) instead of silently picking up stale context
* `--test-run` early stops emit `WORKFLOW_COMPLETED` with `Reason=test-run-stopped-at-<target>` so CI harnesses get a clean terminal state to assert on
* `--scope <x>` can be used mid-workflow to change scope in place. The stage plan re-derives and skipped stages flip to `[S]` if the new scope excludes them
* Welcome message now renders at session start via `settings.json` `companyAnnouncements` instead of running as a workflow stage. Total stage count dropped from 32 to 31
* Statusline progress bar now tracks the current phase. Previously the bar was workflow-scoped (e.g. 15/31) while the numeric ratio was phase-scoped (e.g. 5/7), so the bar could appear stuck for multiple stage advances
* Invalid `AWS_AIDLC_DEFAULT_SCOPE` values now error cleanly and exit without creating state. Previously could hang or ask clarifying questions in non-interactive mode
* New reference documentation chapter at `docs/reference/12-state-machine.md` covering the workflow, phase, and stage state machines, the audit event taxonomy, and the same-commit doc/code consistency rule
* Canonical state file schema is documented as State Version 5. Audit log consumers that parse free-form markdown headings (`## Artifact Update`, `## Subagent Completed`) must switch to the `**Event**:` field shape — these sections are no longer emitted
* Audit event taxonomy rebalanced: 5 auto-event types removed (test-run selections now tag canonical events with `Test-Run: true`), session lifecycle events now emitted by hooks (SessionStart / SessionEnd / PreCompact), artifact lifecycle events now emitted by the PostToolUse hook. Taxonomy: 47 → 41 events, each with a tool-owned emitter
